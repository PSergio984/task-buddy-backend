# Plan 03-01: Automated Audit Logging

## Objective
Implement a reusable decorator to automatically log mutations (create, update, delete) in the audit log.

## Context
Currently, audit logs are supported by models and CRUD helpers but are not being populated. Manually adding `create_audit_log` calls to every CRUD or router function is error-prone. A decorator will standardize this process.

## Tasks

### 1. Create Audit Decorator
- [ ] Create `app/libs/audit.py` (or similar) to house the decorator.
- [ ] Implement `audit_log` decorator that:
    - Accepts `action` (AuditAction) and `target_type` (str).
    - Extracts `db` (AsyncSession) and `user_id` from function arguments.
    - Captures the result of the decorated function.
    - Creates an audit log entry using the result's `id` (if applicable).
    - Handles exceptions (optional: log failures).

### 2. Apply Decorator to Task CRUD
- [ ] Apply `@audit_log` to `create_task`, `update_task`, `delete_task` in `app/crud/task.py`.
- [ ] Apply to `create_subtask`, `update_subtask`, `delete_subtask`.

### 3. Apply Decorator to Project CRUD
- [ ] Apply `@audit_log` to `create_project`, `update_project`, `delete_project` in `app/crud/project.py`.

### 4. Apply Decorator to Tag CRUD
- [ ] Apply `@audit_log` to `create_tag`, `delete_tag` in `app/crud/tag.py`.

### 5. Verification
- [ ] Create a test `tests/routers/test_audit_automation.py`.
- [ ] Perform a mutation (e.g., create a task) via API.
- [ ] Verify that an audit log entry appeared in the database.
- [ ] Verify that the entry has the correct `user_id`, `action`, and `target_id`.

## Success Criteria
- Mutation functions in CRUD layers are decorated.
- Audit logs are automatically populated on every successful mutation.
- No performance regression (using `db.flush()` within the existing transaction).
- Type safety and IDE support for the decorator.
