### Task 1: Add production guard to seed.py

<read_first>
- scripts/seed.py
</read_first>

<action>
Already implemented in this plan's execution:

```python
env = os.environ.get("APP_ENV", "development").lower()
seed_allowed = os.environ.get("SEED_ALLOWED", "").lower()
if env == "production" and seed_allowed != "true":
    logger.error(
        "Seed script blocked in production. "
        "Set SEED_ALLOWED=true to override (destructive — wipes demo user data)."
    )
    raise SystemExit(1)
```

Set `APP_ENV=production` in Render environment variables for the production service.
The seed runs on every startup via `start.sh` — without the guard, any prod restart
wipes the demo user's tasks. With the guard, prod restarts are safe.
</action>

### Task 2: Harden start.sh error handling

<read_first>
- start.sh
</read_first>

<action>
Change the silent-fail fallback for the `alembic_version` clear step from:

```bash
python -c "..." || echo "Warning: could not clear alembic_version ..."
```

to:

```bash
python -c "..." || { echo "FATAL: could not clear alembic_version — check DATABASE_URL"; exit 1; }
```

This ensures a misconfigured DATABASE_URL surfaces as a hard failure rather than
silently proceeding to `alembic stamp head` with a broken connection.
</action>

<acceptance_criteria>
- `start.sh` fallback block uses `|| { echo ...; exit 1; }` (not `|| echo ...`)
- `start.sh` does not contain `|| echo "Warning: could not clear alembic_version"`
</acceptance_criteria>
